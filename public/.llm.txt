# Digital Crew · Max Agent API
# Source: https://docs.digitalcrew.tech/
# Generated from: digitalcrew-docs/public/openapi/max-agent-v1.yaml

## Overview

HTTP API for Digital Crew **Max Agent** — an AI-powered outbound sales automation platform.
Covers: campaigns, prospects, organizations, prospect lists, connected accounts (Unipile),
Unibox (unified inbox), Apollo integration, AI agent, billing, and workspace tooling.

Production base URL: https://max.digitalcrew.tech
Local dev base URL:  http://localhost:3006

---

## Authentication

All authenticated endpoints use HTTP Bearer:
  Authorization: Bearer <token>

Two token types are accepted:

1. Workspace API key — prefixed `max_live_`. Has a fixed set of scopes.
   - 403 Forbidden (with `requiredScope`) if key lacks required scope.

2. Supabase session JWT — three dot-separated base64url segments. Used by the web app.
   - Subject to workspace membership, not scopes.
   - JWT-only routes reject API keys with 401.

The server detects auth type by token shape: three dots → JWT, `max_live_…` → API key.

- Paginated list:    { "data": [...], "count": number, "page": number, "pageSize": number }

## HTTP Status Codes

| Status | Meaning                                                      |
|--------|--------------------------------------------------------------|
| 200    | Success                                                      |
| 201    | Created                                                      |
| 400    | Invalid request body or query parameter                      |
| 401    | Missing, malformed, or expired bearer token                  |
| 402    | Insufficient credits (AI agent / Apollo billing)             |
| 403    | Valid token but missing scope — body includes requiredScope   |
| 404    | Resource not found or not visible to this workspace          |
| 409    | State conflict (e.g. launching a non-draft campaign)         |
| 422    | Semantically invalid (duplicate email, AI provider error)    |
| 429    | Rate limited (reserved — not currently emitted)              |
| 500    | Unexpected server error                                      |
| 502    | Upstream provider unavailable (Unipile, Apollo, OpenAI)      |
| 503    | Feature disabled (e.g. OpenAI off in this environment)       |
| 504    | Upstream provider timeout                                    |

## Pagination

- Query params: `page` (≥ 1, default 1), `pageSize` (1–100, default 20)
- Response includes: `count` (total rows), `page`, `pageSize`
- No Link header — derive: `hasMore = page * pageSize < count`

---

## Rate Limiting

No HTTP rate limits currently. Per-account sending limits are managed by
account-rate-limits endpoints and apply at workflow execution time.

Future: 429 with Retry-After header + X-RateLimit-* headers.

---

## Async Operations

Two flows are async (return immediately, finish in background):

1. POST /api/v1/apollo/people/create-list
   - Creates prospect_list with status: "pending"
   - Cron runs every minute to process; status advances: pending → processing → completed (or failed)
   - Poll GET /prospect-lists/{id} — watch: status, processed_results, total_results, error_message

2. POST /api/v1/campaigns/{id}/launch
   - Creates workflow_execution per prospect, then returns
   - Poll GET /campaigns/{id}/stats and GET /campaigns/{id}/lead-analytics for progress

### CAMPAIGNS
Scope: campaigns:read / campaigns:write
=============================================================================

GET    /api/v1/campaigns
  List campaigns (paginated). Scope: campaigns:read
  Query params: page, pageSize, search (substring on name), status[] (multi),
    sortBy (name|status|created_at|updated_at|started_at|last_activity_at, default: created_at),
    sortOrder (asc|desc, default: desc)
  Response: PaginatedEnvelope<Campaign>

POST   /api/v1/campaigns
  Create campaign in draft state. Scope: campaigns:write
  Required body: { name, included_lists (non-empty), accounts (non-empty) }
  Optional body: description, workflow_config, excluded_lists, exclusion_settings,
    scheduling_config, execution_engine
  Response 201: { data: Campaign }

GET    /api/v1/campaigns/{id}
  Get campaign by UUID. Scope: campaigns:read
  Response: { data: Campaign }

PATCH  /api/v1/campaigns/{id}
  Partial update. Scope: campaigns:write
  Editable: name, description, workflow_config, included_lists, excluded_lists,
    exclusion_settings, scheduling_config, accounts
  Note: use dedicated state-transition endpoints instead of sending status directly.

DELETE /api/v1/campaigns/{id}
  Permanently delete campaign + all workflow_executions. Scope: campaigns:write
  Prefer /archive for soft removal. Response: { success: true }

POST   /api/v1/campaigns/{id}/launch
  Transition draft → active. Scope: campaigns:write
  Creates one workflow_execution per resolved prospect. Returns 409 if not draft.
  Response: CampaignTransitionResponse (includes prospects_added, executions_created)

PATCH  /api/v1/campaigns/{id}/pause
  Transition active → paused. Scope: campaigns:write
  Worker stops dequeueing; in-flight calls not interrupted. 409 if not active.

PATCH  /api/v1/campaigns/{id}/resume
  Transition paused → active. Scope: campaigns:write
  409 if not paused.

PATCH  /api/v1/campaigns/{id}/stop
  Transition active|paused → stopped. Scope: campaigns:write
  Cannot be resumed. 409 if not active/paused.

PATCH  /api/v1/campaigns/{id}/archive
  Transition any → archived. Scope: campaigns:write
  Hidden from default list views but restorable.

PATCH  /api/v1/campaigns/{id}/restore
  Transition archived → draft. Scope: campaigns:write
  409 if not archived.

GET    /api/v1/campaigns/{id}/stats
  Aggregate performance stats. Scope: campaigns:read
  Eventually consistent (~1 minute lag). No date filtering.
  Response: CampaignPerformanceStats
    execution: { total, pending, running, paused, waiting, completed, failed, stopped }
    email: { sent, opened, clicked, replied, bounced }
    linkedin: { invitations_sent, invitations_connected, messages_sent, messages_replied,
      messages_read, inmail_sent, inmail_replied, inmail_read }

GET    /api/v1/campaigns/{id}/lead-analytics
  Per-prospect breakdown in the workflow. Scope: campaigns:read
  Query params: page, pageSize
  Response: paginated per-lead rows with execution_status, message event counters.

GET    /api/v1/campaigns/{id}/node-run-counts
  Map of nodeId → execution count. Scope: campaigns:read
  Useful for funnel visualization on the canvas.
  Response: { nodeRunCounts: { [nodeId]: number } }

### APOLLO
People-search → prospect-list ingestion (async).
Scope: prospect-lists:write
=============================================================================

POST   /api/v1/apollo/people/create-list
  Create Apollo-backed prospect list (async). Scope: prospect-lists:write
  Creates prospect_list with status:"pending", then cron processes every minute.
  Status progression: pending → processing → completed (or failed after 3 retries).
  Poll GET /prospect-lists/{id} for progress.
  Body: {
    list_name: string,
    apollo_search_payload: object,  // Apollo mixed_people/search body
    idempotency_key?: string        // safe to retry with same key
  }
  Example apollo_search_payload:
    { person_titles: ["Founder","CEO"], organization_locations: ["United States"],
      organization_num_employees_ranges: ["1-10","11-50"] }
  Response: { data: ProspectList } (status: "pending")

POST   /api/v1/apollo/people/add-more
  Append more leads to existing Apollo list (async). Scope: prospect-lists:write
  Re-runs saved apollo_search_payload to pull next N results.
  Body: { list_id: uuid, count?: int (1-10000, default: 100) }
  Response: { data: { list: ProspectList, message: string } }

### API KEYS
JWT only — create/list/revoke workspace API keys.
=============================================================================

GET    /api/v1/api-keys
  List API keys (secrets masked). JWT only.
  key_prefix field shows first chars of secret (e.g. max_live_a1b2).
  Response: { data: ApiKeyListItem[] }

POST   /api/v1/api-keys
  Create API key — plaintext secret returned ONCE. JWT only.
  Store secret immediately; subsequent reads only return prefix.
  Body: { name: string, scopes: Scope[] (minItems: 1) }
  Response 201: { data: ApiKeyCreated } (includes key: "max_live_<32chars>")

DELETE /api/v1/api-keys/{id}
  Revoke API key (immediate). JWT only.

--- Key Format ---
max_live_<32 random alphanumeric characters>

### WORKFLOW CONFIG
=============================================================================

WorkflowConfig:
  nodes: WorkflowNode[] (minItems: 1)
  edges: WorkflowEdge[]
  ai_message_reviews: { [nodeId]: { [prospectId]: { subject?, body?, message? } } }

WorkflowNode:
  id: string
  type: campaignStart | action | condition | delay | wait | label | end
  position: { x: number, y: number }
  data: object (shape varies by type)

  For type "action", data.action_type is one of:
    send_email
    send_linkedin_message
    send_linkedin_inmail
    send_linkedin_invitation
    view_profile
    view_chat

WorkflowEdge:
  id: string
  source: string (node id)
  target: string (node id)
  sourceHandle?: string (e.g. "success", "timeout", "true", "false")
  targetHandle?: string
  type?: string
  data?: object

### SETUP GUIDE
=============================================================================

1. Get a workspace API key
   - Log into the web app (JWT session required)
   - POST /api/v1/api-keys with desired name and scopes
   - Copy the returned key (shown once only) — format: max_live_...

2. Set the Authorization header for all subsequent requests:
   Authorization: Bearer max_live_<your_key>

3. Connect a sending account (LinkedIn or email)
   - POST /api/v1/unipile/hosted-auth/link with type:"create"
   - Open the returned URL and authenticate
   - Account appears in GET /api/v1/accounts once connected

4. Import prospects
   - POST /api/v1/prospects/bulk-import — for CSV-style data
   - POST /api/v1/apollo/people/create-list — for Apollo people search (async)
   - POST /api/v1/prospect-lists/import-csv — for CSV with list creation

5. Create a prospect list
   - POST /api/v1/prospect-lists — empty list, then add members
   - Or import-csv creates list + members in one call

6. Create a campaign
   POST /api/v1/campaigns with:
   {
     "name": "My Campaign",
     "included_lists": ["<list_uuid>"],
     "accounts": [{ "account_id": "<account_uuid>" }],
     "workflow_config": { "nodes": [...], "edges": [...] }
   }

7. Launch the campaign
   POST /api/v1/campaigns/{id}/launch
   - Campaign transitions draft → active
   - Worker processes executions out of band

8. Monitor progress
   - GET /api/v1/campaigns/{id}/stats — aggregate counters
   - GET /api/v1/campaigns/{id}/lead-analytics — per-prospect status
   - GET /api/v1/unibox/chats — incoming replies

=============================================================================
### ERROR HANDLING
=============================================================================

All errors:
  { "error": "human-readable message", "details": { ... } }

Validation (400):
  { "error": "Invalid request body", "details": { "fieldErrors": { "field": ["message"] } } }

Scope error (403):
  { "error": "Forbidden", "requiredScope": "campaigns:write" }

Conflict (409):
  { "error": "Campaign cannot be launched from current status", "details": { "current_status": "active" } }

Insufficient credits (402):
  { "error": "Insufficient credits", "details": { "required": 100, "balance": 12 } }

Upstream provider (502):
  { "error": "Upstream provider error", "details": { "provider": "unipile" } }

=============================================================================
### DEDUPLICATION RULES
=============================================================================

Prospects: deduplicated by normalized email within workspace
  - Duplicate on POST /api/v1/prospects → returns existing row with HTTP 200
  - Bulk import: duplicates counted in "existing"

Organizations: deduplicated by normalized primary_domain
  - Normalization: lowercase, strip scheme (http/https), strip www., strip trailing slash

=============================================================================
### CREDITS SYSTEM
=============================================================================

Credits are consumed by:
  - Apollo people search (per lead ingested)
  - AI workflow generation
  - AI message preview generation
  - Unipile daily account charge

Check balance: GET /api/v1/billing/workspace-balance (JWT only)
Quote Apollo cost: GET /api/v1/billing/apollo-search-quote?maxLeads=N (JWT only)
View ledger: GET /api/v1/billing/workspace-movements (JWT only)

Endpoints return 402 when credits are insufficient.
